# Web 住宅设计工具可行性与产品设计

更新日期：2026-04-26

## 一句话结论

这个项目可行，但不应该做成“线稿自动转 3D”，也不应该做成“简化版 CAD”。更合理的方向是：**面向自建房业主的轻量住宅结构建模与外观沟通工具**。

用户在 2D 中用很少的住宅专用工具编辑墙、窗、楼层、开孔、阳台、屋顶和材质；系统底层保存结构化住宅对象；Three.js 根据这些对象实时生成 3D 外观预览；imagegen 负责生成真实材质、局部装饰贴片和效果图增强，但不负责决定房屋几何结构。

第一版成功标准不是直接出施工图，而是让业主能快速建立一个尺寸基本可信、外观表达清楚、可反复修改的三层别墅模型，用于和设计师沟通。

---

## 1. 产品定位

### 1.1 目标用户

核心用户是不会 CAD、但需要和设计师高效沟通的自建房业主。这个用户需要的是“把脑子里的房子搭出来”，而不是学习一套专业制图软件。

这个工具要服务以下场景：

- 表达三层别墅的整体布局、楼层关系和外观体量
- 修改门窗、阳台、楼层高度、墙面开孔和屋顶样式
- 快速预览不同外墙、屋顶、窗框、栏杆材质效果
- 导出 2D/3D 截图或说明材料，与设计师讨论

### 1.2 不做什么

第一版明确不做：

- 施工图出图
- 结构安全计算
- 水电暖管线设计
- 自动报规和规范审查
- 复杂地形和场地工程
- 室内精装修全套设计
- 任意自由曲面建筑
- CAD 式通用绘图命令系统

这些能力可以在未来接入专业工作流，但不应进入第一版核心范围。

## 2. 核心产品原则

### 2.1 简单，但不是随便画

用户表面上是在画线和调整元素，但底层必须是结构化对象。墙不是普通线段，窗不是普通矩形，阳台不是装饰图形。它们都必须是可约束、可测量、可投影、可生成 3D 的住宅对象。

这能避免一个常见失败模式：2D 草图很好看，但生成 3D 时没有厚度、没有楼层关系、窗洞无法落在墙上、立面和平面互相矛盾。

### 2.2 强约束优先

第一版选择强约束模型：系统不允许用户画出无法生成 3D 的结构。

典型规则：

- 窗、门、开孔必须依附在墙上
- 墙必须属于某个楼层
- 阳台必须依附楼板边缘或外墙
- 楼层高度变化必须同步影响所有立面和 3D
- 立面编辑不能制造与平面墙体不存在的外墙
- 屋顶必须基于已有外轮廓生成

强约束会牺牲少量自由度，但能换来稳定生成、可反复编辑和低学习成本。

### 2.3 2D 是编辑入口，3D 是实时结果

顶部只需要两个主模式：

- `2D`
- `3D`

在 `2D` 模式下，用户选择具体视图：

- 1F 平面
- 2F 平面
- 3F 平面
- 正立面
- 背立面
- 左立面
- 右立面
- 屋顶视图

这些视图不是互相独立的图纸，而是同一栋房子的不同投影。用户在任一视图中编辑，最终都修改同一套底层住宅对象。

### 2.4 imagegen 负责真实感，不负责结构

imagegen 很适合生成：

- 外墙涂料、真石漆、石材、文化石纹理
- 屋瓦、金属屋面、木纹、混凝土纹理
- 阳台栏杆、窗套、腰线、檐口、格栅等 facade sprite
- 基于 3D 截图的后期效果图增强

但 imagegen 不应该负责：

- 判断墙体几何
- 生成窗洞结构
- 决定楼层关系
- 推断屋顶坡度
- 生成可编辑的建筑模型

结构必须由程序生成，AI 图像只作为外观层附着在结构上。

## 3. 推荐方案：多视图结构编辑器

### 3.1 总体方案

推荐采用“多视图结构编辑”：

用户可以在平面和立面里编辑墙、窗、楼层、开孔、阳台和屋顶，但所有编辑都落到同一套 `HouseModel` 上。Three.js 根据 `HouseModel` 实时生成 3D 外观。

这个方案的关键不是渲染，而是统一数据模型和约束系统。

### 3.2 为什么不选其他方案

#### 平面优先方案

只画每层平面，然后自动生成 3D。

优点是实现最快，适合早期户型探索。缺点是立面控制弱，难以满足用户对门窗、开孔、阳台、楼层和外观的直接编辑需求。

#### 效果图优先方案

先生成一个大概 3D，再让 imagegen 根据截图生成漂亮外观。

优点是视觉效果快。缺点是不可控、不可精确、不可稳定复改。每次生成的外观可能漂移，不适合作为设计沟通的结构依据。

#### 多视图结构编辑方案

平面和立面都能编辑，但编辑的是同一套结构对象。

优点是既能保持尺寸可信，又能满足外观编辑。缺点是约束系统和数据模型更复杂。考虑到目标是三层别墅沟通工具，这是最值得做的方向。

## 4. 系统架构

### 4.1 模块划分

建议把系统拆成七个核心模块：

1. `HouseModel`
   - 保存房屋结构数据
   - 包含楼层、墙体、开孔、门窗、阳台、屋顶、材质等对象

2. `ConstraintEngine`
   - 校验编辑是否合法
   - 负责吸附、对齐、依附关系、尺寸限制和冲突提示

3. `ViewProjection`
   - 把同一套结构对象投影到不同 2D 视图
   - 支持平面、立面和屋顶视图

4. `EditTools`
   - 提供少量住宅专用工具
   - 包括墙、门、窗、开孔、阳台、楼层高度、屋顶、材质刷

5. `GeometryBuilder`
   - 把结构对象转换成 Three.js 几何
   - 负责墙体厚度、门窗洞口、楼板、阳台、屋顶等模型生成

6. `MaterialSystem`
   - 管理贴图、材质、sprite 和 facade patch
   - imagegen 生成的图片通过这里进入 Three.js

7. `Renderer3D`
   - Three.js 场景、相机、灯光、阴影、材质预览和截图导出

### 4.2 数据流

标准编辑流程：

1. 用户在 2D 视图执行编辑
2. `EditTools` 把操作转换成结构对象变更
3. `ConstraintEngine` 校验是否合法
4. 合法变更写入 `HouseModel`
5. `ViewProjection` 更新所有 2D 视图
6. `GeometryBuilder` 重建受影响的 3D 几何
7. `Renderer3D` 实时显示结果

所有视图都必须从 `HouseModel` 派生，不能各自保存一份独立图形数据。

## 5. 核心数据模型

### 5.1 Project

项目级数据：

- 项目名称
- 单位系统
- 默认墙厚
- 默认层高
- 场地朝向
- 材质库
- 当前选中视图

### 5.2 Storey

楼层对象：

- 楼层编号：1F、2F、3F
- 标高
- 层高
- 楼板厚度
- 外轮廓
- 所属墙体集合

第一版固定支持三层，但内部模型不应写死三层，方便后续扩展。

### 5.3 Wall

墙体对象：

- 起点和终点
- 厚度
- 高度
- 所属楼层
- 外墙/内墙类型
- 材质
- 依附的 openings

墙体是 3D 生成的主骨架。

### 5.4 Opening

开孔对象：

- 所属墙体
- 类型：门、窗、空洞
- 距墙起点位置
- 离地高度
- 宽度
- 高度
- 样式
- 窗框/门框材质

门窗本质上都是墙上的 opening，只是表现样式不同。

### 5.5 Balcony

阳台对象：

- 所属楼层
- 依附墙体或楼板边缘
- 宽度
- 进深
- 栏杆高度
- 底板厚度
- 栏杆材质
- 侧边封闭方式

第一版只支持矩形阳台，优先覆盖真实自建房中最常见的阳台形态。

### 5.6 Roof

屋顶对象：

- 屋顶类型：平屋顶、单坡、双坡
- 坡度
- 檐口外挑
- 屋脊方向
- 屋面材质

第一版不做复杂多坡屋顶和异形屋面。

### 5.7 Material 与 FacadePatch

材质分两类：

- `Material`：可平铺贴图，例如外墙漆、石材、屋瓦、木纹
- `FacadePatch`：局部装饰贴片，例如窗套、腰线、格栅、栏杆图案

imagegen 生成的素材应进入这两类对象，而不是直接覆盖整个建筑。

## 6. 2D 编辑体验

### 6.1 顶部模式切换

顶部只放核心模式：

- `2D`
- `3D`

用户不应先面对复杂工具栏。进入 `2D` 后再选择具体视图。

### 6.2 2D 视图切换

2D 下的视图使用清晰标签：

- `1F`
- `2F`
- `3F`
- `正面`
- `背面`
- `左侧`
- `右侧`
- `屋顶`

平面图用于编辑墙体布局、门窗位置、阳台平面位置。

立面图用于编辑：

- 窗和门的离地高度
- 开孔高度
- 阳台高度和栏杆
- 楼层高度
- 屋檐、屋顶坡度
- 外墙材质分区

### 6.3 工具集合

第一版工具栏应控制在少量住宅专用工具：

- 选择/移动
- 墙
- 门
- 窗
- 开孔
- 阳台
- 楼层高度
- 屋顶
- 材质刷
- 尺寸标注

避免 CAD 式命令系统，例如偏移、修剪、延伸、镜像、阵列等。必要能力可以做成上下文操作，而不是显式命令。

### 6.4 尺寸输入

每次绘制或选中对象时，侧边属性面板显示关键尺寸：

- 墙长
- 墙厚
- 窗宽
- 窗高
- 离地高度
- 阳台宽度
- 阳台进深
- 层高
- 屋顶坡度

用户可以拖动，也可以输入精确数值。这样比 CAD 简单，但仍能保持尺寸基本准确。

## 7. Three.js 3D 预览

### 7.1 3D 目标

3D 预览的目标是让用户看清：

- 三层体量关系
- 墙体比例
- 门窗位置
- 阳台外观
- 屋顶形式
- 材质效果
- 外观整体风格

第一版不追求照片级实时渲染，而是追求稳定、清晰、可反复修改。

### 7.2 渲染内容

第一版 3D 应包含：

- 墙体几何
- 楼板和层间关系
- 门窗洞口
- 简化门窗模型
- 阳台底板和栏杆
- 平屋顶/坡屋顶
- 外墙材质
- 屋顶材质
- 基础日光、环境光、阴影
- 轨道相机
- 正面/背面/左侧/右侧快捷视角

### 7.3 几何生成策略

墙体、楼板、阳台和屋顶建议由程序生成网格。门窗开孔可以先采用稳妥的墙面分割策略，而不是一开始就追求复杂布尔运算。

推荐优先级：

1. 先把墙体按 opening 切分为多个矩形面片
2. 再生成厚度和侧边
3. 最后加门窗框、玻璃和局部装饰

这样比实时布尔更容易调试，也更适合住宅规则化几何。

## 8. imagegen 使用策略

### 8.1 材质库为主

第一版应优先生成和维护一个材质库：

- 白色外墙涂料
- 米白真石漆
- 灰色石材
- 文化石
- 深灰屋瓦
- 红棕屋瓦
- 木纹格栅
- 黑色金属栏杆
- 白色窗框
- 深灰窗框

这些素材作为可复用贴图进入 Three.js。

### 8.2 立面贴片为辅

对于不适合建成复杂几何的装饰，可使用 facade sprite：

- 窗套
- 腰线
- 檐口线脚
- 格栅
- 局部石材拼花
- 栏杆图案

贴片必须依附在墙面或阳台上，并记录位置、尺寸和所属对象。

### 8.3 效果图增强作为后期能力

未来可以提供“生成效果图”按钮：

1. Three.js 输出当前视角截图
2. 用户选择风格关键词
3. imagegen 生成更真实的外观效果图

这张图只能作为展示和沟通材料，不应反向覆盖结构模型。

## 9. 可行性评估

### 9.1 技术可行性

整体技术可行。Three.js 足以承担住宅体量、门窗、阳台、屋顶、材质和实时预览。Web 技术也足以承载 2D 编辑器、属性面板、项目保存和导出。

真正的难点不在渲染，而在：

- 数据模型是否统一
- 平面和立面是否能同步编辑
- 约束系统是否足够清晰
- 几何生成是否稳定
- UI 是否足够简单

### 9.2 产品可行性

产品方向成立，因为目标不是替代设计师，而是改善业主与设计师之间的沟通。这个定位比“业主自己完成专业设计”更现实，也更容易做出有价值的第一版。

### 9.3 最大风险

最大风险是范围失控。如果试图同时做 CAD、BIM、AI 生成、施工图、室内设计、效果图和规范校验，项目会迅速变得不可控。

第一版必须坚守“外观沟通工具”边界。

- 楼梯细节

## 11. 分阶段路线

### 阶段 1：结构模型和最小 3D 闭环

目标：证明“结构对象 -> Three.js 房子”可行。

范围：

- Project、Storey、Wall、Opening 基础模型
- 单层平面墙体绘制
- 墙体厚度和高度
- 窗洞/门洞
- Three.js 基础生成
- JSON 保存和加载

成功标准：画一个简单矩形房子，加门窗，能稳定生成 3D。

### 阶段 2：三层和平立面同步

目标：证明多视图结构编辑可行。

范围：

- 1F、2F、3F
- 楼层高度
- 四个立面视图
- 在立面中编辑窗高、开孔、楼层高度
- 所有视图同步

成功标准：在平面和立面中编辑同一个窗或楼层，3D 实时一致。

### 阶段 3：阳台、屋顶和材质库

目标：让模型接近真实别墅外观。

范围：

- 矩形阳台
- 平屋顶/双坡屋顶
- 外墙、屋顶、窗框材质
- imagegen 材质导入
- 3D 快捷视角和截图

成功标准：能搭出一个有门窗、阳台、屋顶和材质的三层别墅外观。

### 阶段 4：沟通交付能力

目标：让工具真正服务设计沟通。

范围：

- 导出 2D 视图截图
- 导出 3D 视角截图
- 给对象添加备注
- 生成一页方案说明
- 可选：基于 3D 截图生成效果图

成功标准：用户能把导出的材料发给设计师讨论。

## 12. 推荐技术栈

### 12.1 前端

- React 或 Vue：承载应用界面
- TypeScript：保证结构对象类型可靠
- SVG 或 Canvas：实现 2D 编辑器
- Three.js：实现 3D 预览
- 本地 JSON：第一版项目保存格式

### 12.2 2D 编辑器选择

第一版更建议从 SVG 或 Canvas 自研轻量编辑层开始，而不是直接套用通用 CAD 库。原因是本项目的对象很少，规则明确，自研能避免把 CAD 的复杂度带进产品。

平面/立面视图本质是结构对象投影，所以 2D 层只需要负责显示、选中、拖动、吸附和输入，不需要变成完整矢量编辑器。

### 12.3 3D 生成

Three.js 中应把每类结构对象映射成独立生成器：

- `buildWallMesh`
- `buildOpeningFrame`
- `buildStoreySlab`
- `buildBalcony`
- `buildRoof`
- `applyMaterial`

这样后续可以逐步替换内部实现，而不影响上层编辑器。

## 13. 关键风险与规避

### 13.1 多视图冲突

风险：平面和立面编辑同一对象，容易出现矛盾。

规避：所有编辑都写入同一套结构对象。2D 图形永远只是投影，不保存独立真相。

### 13.2 用户觉得还是像 CAD

风险：工具太多、术语太专业、学习成本太高。

规避：工具栏只保留住宅对象。复杂操作做成属性面板和上下文按钮。避免 CAD 命令词。

### 13.3 3D 生成不稳定

风险：墙洞、阳台、屋顶组合后出现破面和错位。

规避：第一版限制几何形态。墙体开孔先用面片切分，不依赖复杂实时布尔。

### 13.4 imagegen 结果不可控

风险：AI 生成的外观漂亮但不稳定，影响可编辑性。

规避：imagegen 只生成材质和贴片，不生成结构。效果图增强只能作为导出结果。

### 13.5 范围膨胀

风险：过早加入室内、施工图、规范和云协作。

规避：第一版只围绕“三层别墅外观沟通”验收。

---

## 14. 最小可验证原型

最小原型建议只做一个闭环：

1. 在 1F 平面画一个矩形外墙
2. 输入墙厚和层高
3. 在一面墙上添加一个窗
4. 切到正立面，调整窗的离地高度
5. 切到 3D，看到带窗洞的墙体
6. 给外墙换一个材质
7. 保存项目 JSON
8. 重新加载后结构不丢失

如果这个闭环稳定，说明底层路线正确。之后再扩展三层、阳台、屋顶和多立面。

---

## 15. 最终建议

建议把项目正式定义为：

**Web 轻量住宅建模与外观沟通工具**

第一版聚焦：

- 三层别墅
- 结构化 2D 多视图编辑
- 强约束对象模型
- Three.js 实时 3D 外观预览
- imagegen 材质库和少量 facade sprite
- 面向设计师沟通的导出能力

这条路线技术上可行，产品上也有明确价值。它避开了 CAD 和 BIM 的复杂度，也避免了纯 AI 效果图不可控的问题。核心判断是：**几何必须结构化，外观可以 AI 增强；2D 负责编辑，3D 负责验证和沟通。**
